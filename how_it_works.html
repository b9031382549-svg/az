Page 2 · The six stations, in detail

What actually happens inside each box

1MEMORY — the confirmed-answer store

Before any AI runs, the line is checked against everything the system has ever answered and had confirmed. The lookup ignores cosmetic differences — spacing, capitalisation, punctuation — so “Coca-Cola 1L” and “COCA COLA 1 L” hit the same record.

Where memory entries come from: answers confirmed by the agreement gate, research results with citations, human decisions, and the double-verified reference set. Memory only ever contains confirmed knowledge — a plain guess can never enter it.

In: raw product name · Out: the confirmed code, instantly — or “not seen, pass on”
zero costinstantgrows every day

2SERVICE CHECK — goods or not?

Invoices are full of lines that are not products at all: transport fees, rent, repair works, utility bills, subscription charges, even letter references and dates. These must get no commodity code. A dedicated detector — patterns and vocabulary tuned on your real invoices, plus the AI’s own judgement — separates them out with over 99% reliability.

In: unseen line · Out: “Service” (done) — or “this is a good, continue”
99%+ measuredprotects everything downstream

3LOCAL AI — the reader

A compact language model that we trained specifically on this task, using thousands of worked examples with written reasoning, built from your own invoice data. It decodes what a line actually means — brands, Azerbaijani and Russian words, transliterations, abbreviations, size codes — decides what the real product is, and proposes the 4-digit code with a short written justification.

It runs entirely on your own hardware. One line takes about a second; many copies run side by side for volume. No product name leaves the building at this step.

In: product name · Out: proposed code + one-paragraph justification
~83% correct alone — before the safety layersruns on your servers

4AGREEMENT GATE — the certainty test

The key discovery of this project: an AI’s own “confidence” is worthless — models feel equally sure when right and when wrong. What actually predicts correctness is agreement between independent opinions. So the proposal from step 3 is checked against opinions that were formed differently:

· the similarity search — do the nearest already-confirmed products carry the same code? (page 3 explains how)
· a second AI of a different design, reasoning from scratch.

All agree → the answer is released; measured ≥95% correct on everything released this way, with zero service/product mix-ups. Any disagreement → the line is escalated, never guessed.

In: proposal + independent opinions · Out: released answer — or escalation
≥95% on released answers0 wrong-type leaks measured

5WEB RESEARCH — the evidence hunt

Runs only for the minority the models disagree on — typically cryptic brand names. The system searches the open web for what the product really is and may resolve the line only if it can cite the evidence. No citation — no answer. Page 4 walks through the process with real cases.

In: disputed line · Out: answer + source link — or “to human review”
88% of disputes resolved~96% correctevery answer cited

6HUMAN EXPERT — the honest residue

The few percent that survive every automatic step. The reviewer doesn’t start from zero: each line arrives with the AI’s analysis and a pre-written question that names the exact missing fact — e.g. “Is this cream sold as a cosmetic or as a registered medicine?” One click to answer.

The answer is written into MEMORY permanently. The same question is never asked twice.

In: unresolved line + a targeted question · Out: decision → memory, forever
~2.5% of volumeshrinks every month
Page 3 · Deep dive — the similarity search (“vectors”)

How the system finds “products like this one”

Computers can’t compare meanings — only numbers. So every product name is converted into a meaning fingerprint: a long list of numbers computed by a language model, built so that names with similar meaning get similar numbers — even when the spelling, language or word order is completely different. “Coca-Cola 1L”, “koka kola 0,5”, and “Кока-Кола ПЭТ” land practically on top of each other; a timber board lands far away.

The vote. When a new name arrives, the system finds its nearest neighbours among confirmed products only. If several close neighbours all carry the same code, that is an independent opinion: “everything that means this, is coded this.” If the neighbours disagree, or nothing is close enough, the similarity search honestly abstains — it never votes on weak evidence.

Three jobs, one mechanism:

JobHow similarity search does it
Second opinion in the agreement gatenearest confirmed neighbours vote on the proposed code
Collapse duplicates“Fairy 450ml”, “FAIRY qabyuyan 0.45” — one family, answered once
Route the rare stuffa name with no close neighbour is flagged “genuinely new” early
Why it strengthens itself: the neighbour map is drawn over confirmed answers only. Every month the map gets denser, so more new names land next to solid neighbours — the free layer keeps taking work away from the paid ones. Today the confirmed map holds ~1,300 double-verified products and everything the flywheel confirms on top.
Page 4 · Deep dive — how web research works

What happens when the models disagree

1 · build search queries→ 2 · read real sources→ 3 · identify the product→ 4 · apply the rulebook→ 5 · answer + citation→ no evidence? → human

1 — Queries. The researcher builds several searches from the line: the brand alone, the brand + product words, transliterations (Azerbaijani ↔ Russian ↔ English), and any model/part numbers.

2 — Sources. It reads what comes back: manufacturer pages, retailer listings, pharmacy registries, parts catalogues. Marketing pages are treated as identification evidence, not as classification authority.

3 — Identification. The goal is one sentence: what is this thing, really? This is where names that fooled everything upstream get cracked:

“Maqna Nano Carbon Black” — looks like an industrial chemical. A retailer page shows it is a cigarette brand → 2402, not the chemical code. The invoice header even said otherwise; the research step caught the truth.
“SARAVOY ALT CBKD-11” — meaningless until the part number matches a lower ball joint for a Chevrolet Lacetti on a parts site → vehicle part 8708.
“Caramel 101 C1 43-18-125” — the numbers 43-18-125 are a standard lens-bridge-temple measurement → eyeglass frames 9003, not food.

4 — The rulebook. With the product identified, the code is chosen using the official international classification rules and their explanatory notes (the same book customs officers use) — e.g. a vitamin sold as a food supplement goes to 2106, not to medicines; alcohol-free beer goes to soft drinks 2202.

5 — The evidence rule. The step may answer only if it can attach the source it used. No source → the line goes to a human, with everything the research found already written up. Measured on real disputed lines: 88% resolved, ~96% of those correct.

Privacy: this is the only step where a product name is sent outside your infrastructure (as a web search). It can be restricted to approved sources or switched off entirely — those lines then go straight to human review.
Page 5 · How we know the numbers are true — and why it improves

Measurement, the answer key, and the loop

The answer key. You can’t grade a system without a trusted key. Ours is built from your real invoice data by two unrelated frontier AI systems working independently — one from Anthropic (with web research), one from OpenAI. Only items where both arrive at the same code enter the key; disagreements are excluded and flagged. Today the key holds 1,300+ double-verified products and grows in weekly waves, prioritising the most frequent products on your invoices.

The grading discipline. The model being graded never sees the answer key during training — a held-out slice is kept frozen purely for measurement. Every number on these pages comes from grading against that key, and every upgrade is re-measured before it ships. An upgrade that measures worse is rejected (this has already happened once — the measurement caught it).

The loop, concretely. Every confirmed answer does three things at once: joins MEMORY (that exact name never costs again) · joins the similarity map (its whole neighbourhood gets easier) · joins the next training set for the local AI (the reader itself gets smarter). That’s one confirmation improving three different layers — which is why the expensive tail shrinks month after month without anyone maintaining rules by hand.

What makes it stronger next — in order of impact: using the whole invoice line (supplier, price, VAT, unit — not just the name); a second local AI of a different family so even the offline council is fully independent; and calibrating the gate into a formal, auditable ≥95% guarantee.
